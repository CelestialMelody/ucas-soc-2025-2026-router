# v6：多线程尝试与移除说明

## 1. 为何 PathFinder 主循环不能多线程

PathFinder 每轮迭代的语义是：**按 `net_order` 顺序依次对每条线网做 rip-up 再 route**。

- 第 1 条线网 rip 掉再布，看到的 `occupancy` 是「当前其它线网占好的」。
- 第 2 条再 rip 再布，看到的是「第 1 条已按新路径占好 + 其它线网」。
- 依此类推，**后面的线网必须在前面的线网都更新完 occupancy 之后再布**，冲突才能被逐步消除，overuse 才会下降并收敛。

若把线网分给多个线程**并行** rip+route：

- 各线程同时读写 `occupancy`，每条线网看到的是「部分已更新、部分未更新」的中间状态。
- 等价于**打乱 net_order 的先后关系**，算法假设被破坏，overuse 难以收敛（实测 4 线程时 overuse 震荡不降，单线程约 10 轮收敛到 0）。

**结论**：主循环内「按线网分线程并行 rip+route」与 PathFinder 语义不符，**不能**采用。代码中相关多线程逻辑已移除。

---

## 2. 已移除的多线程相关代码

| 位置 | 移除内容 |
|------|----------|
| `config.hpp` | `ROUTER_NUM_THREADS`、`ROUTER_NUM_THREADS_CXX` 宏/常量 |
| `pathfinder.hpp` | `#include <shared_mutex>`、`#include <thread>`，成员 `occupancy_mutex_` |
| `pathfinder.cpp` | `#include <thread>`、`#include <mutex>`；`routeNetAStar` 中 `#if ROUTER_NUM_THREADS > 1` 的 thread_local A* 网格分支；`ripUpNet` / 写回 occupancy 的 `unique_lock`；`run()` 中按线程数分支的启动信息与并行 worker 循环，改为单一顺序循环 |

当前主流程为**单线程顺序** rip+route，与 v5 行为一致，保证收敛。

## 3. 其它模块是否适合多线程

| 模块 | 是否适合多线程 | 说明 |
|------|----------------|------|
| PathFinder 主循环（rip+route） | 否 | 顺序依赖，见上文。 |
| Cleanup 阶段 | 否 | 同样依赖「当前冲突状态」顺序重布，并行会破坏语义。 |
| 解析（parser） | 否 | I/O + 单次解析，量小，并行收益低。 |
| 输出（output） | 否 | 单次写文件，无并行点。 |
| 多组独立 PathFinder 取最优 | 是 | 若需用多线程，可在外层跑多组不同 `net_order`（如不同随机种子），每组单线程，最后取 wirelength/overuse 最优的一组；当前版本未实现。 |

**结论**：当前代码中**没有**其它适合在保留正确性的前提下做多线程的 hotspot，因此未再引入多线程。
