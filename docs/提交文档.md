# 基板布线程序说明文档

## 任务要求与背景

### 背景

2.5D 异质集成里，芯粒通过基板上的金属互连。布线就是在基板上给每条线网找路径，把引脚连起来。作业把布线区域抽象成规则网格（gcell），在网格图上为线网找路径，要求线网连通、每格最多一根线，并尽量缩短总线长。评测看总线长、过孔数、溢出数、运行时间等。

### 三点要求

1. **读入网表并转成布线用数据**读 `netlist.json`，解析 `net_name`、`net_type`、每条线网的引脚列表（`pins`/`bumps` 里的 `pin_name`/`bump_name`），再和布局里的引脚坐标对上，得到 `Input`（线网列表、每条线的 bumps、pin 输出层名和物理坐标等）。
2. **读入基板 grid**读 `grid_layout.json`，从 `grid_info` 取 `grid_length`、`grid_max_width`、`grid_max_height`，算网格宽高：`width = max_width / grid_length`，`height = max_height / grid_length`。从 `bottom_layer`（C4）、`top_layer`（Chiplet）读引脚名和 `grid_coord_x/y`，转成网格坐标，建“引脚名→坐标”的映射，供网表用。金属层数在 config 里配（如 4 层 M1～M4）。
3. **在基板上完成布线**
   在网格图上用 PathFinder + A* 做 rip-up & reroute，满足每格一根线、无短路断路、无锐角，输出各线网的线段到 JSON。线段按网格、L 形或直线，不产生锐角。

### 约束

- 允许多线程，最多 4 线程。当前实现主循环是单线程（见 4.6 和第 6 节）。
- 禁止：编译/运行失败、改案例设计信息、布线违例、单格多线、锐角、超 1 小时或崩溃。

### 名词说明

后文会反复出现以下术语，在此简要说明：

- **gcell / 粗网格**：将基板按 `grid_length` 划分成的规则单元。物理坐标除以 grid_length 得到网格坐标，布线在网格边上走，即“粗网格”上的全局布线。
- **PathFinder**：一种迭代式全局布线思路：每轮对所有线网先拆线再重布，对历史上发生冲突的格点加“历史代价”，让后续轮次绕开，直到无冲突或达到迭代上限。
- **rip-up（拆线）**：在重布某条线网前，从当前占用表里减去该线网占用的格点，相当于暂时移除这条线的路径。
- **reroute（重布）**：对已拆线的线网用 A* 重新找一条路径并写回占用表。
- **A\***：在网格图上从起点到终点的最短路径搜索算法；本程序用“代价 = 步长 + 历史代价 + 拥塞惩罚”，启发式用曼哈顿距离与层差，在 6 邻域（上下左右+上下层）上扩展。
- **MST（最小生成树）**：多引脚线网不按输入顺序逐段连接，而先用 Prim 算法按引脚间曼哈顿距离求近似最小生成树，得到连接顺序，再在相邻点对之间做 A*，以减小总线长。
- **PagedGrid**：不把整张网格做成一个大数组，而是按 64×64×层 分块，按需分配块；只分配被布线触及的块，大规模测例下内存从“整图×层数”降到与布线规模相关。
- **Cleanup**：PathFinder 主循环到最大迭代仍有余留冲突时，仅针对冲突格和涉及线网做“高代价再拆再布”的收尾阶段，不扫全图。

## 输入输出与运行方式

### 输入

- **grid_layout.json**：`grid_info`（grid_length、grid_max_width/height）；`bottom_layer`（c4_name、grid_coord_x/y）；`top_layer`（bump_name、grid_coord_x/y）。程序用 grid_length 把物理坐标除成网格坐标。
- **netlist.json**：`nets` 里每条线有 net_name、net_type、pins/bumps（pin_name/bump_name）。用名字查 grid_layout 得到网格坐标和层。

### 输出

- **output.json**：线网名为 key，值为线段数组；每段是 start_grid_coordinate、end_grid_coordinate（[x, y, layer_name]，物理坐标）。会从引脚物理坐标接到网格点（L 形），再写网格段。程序还会打 wirelength、disconnected_nets、invalid_segments、grid_overuse、sharp_angles。

### 编译与运行

```bash
mkdir -p build && cd build
cmake ..
cmake --build .
./router <grid_layout.json> <netlist.json> <output.json>
```

示例：

```bash
./router ../benchmark/C2IO1_grid_layout.json ../benchmark/C2IO1_netlist.json ../benchmark/output.C2IO1.json
```

## 整体架构与流程

### 主流程

入口大致如下（`src/main.cpp`）：

```cpp
string grid_content = readFile(grid_file);
string net_content = readFile(net_file);
Input input = parseInput(grid_content, net_content);
PathFinderRouter router(input);
router.run(MAX_PATHFINDER_ITERATIONS);
writeOutput(input, output_file);
```

即：读两个 JSON → 解析成 `Input` → 建 router 并跑 PathFinder → 若未收敛会在 `run()` 里调 Cleanup → 最后写 output.json。

## 算法与数据结构

### 网格与 Key

网格由 grid_length 和 max_width/height 得到；每格 (x, y, layer)。格点用一维 Key 表示，和 config 里一致：

```cpp
// config.hpp
Key calculateCellKey(int x, int y, int l, int grid_width, int grid_height) {
    if (l < 0 || l >= METAL_LAYER_COUNT) return -1;
    return (long long)(l * grid_height + y) * grid_width + x;
}
```

### PathFinder

每轮按 net_order 顺序：对每条线网先 rip（从 occupancy 里减掉该线网），再 A* 重布并写回 occupancy。本轮里 occupancy>1 的格点在轮末加 history_cost，下一轮 A* 就会绕开。这样迭代直到无冲突或到最大轮数；若还没消完就进 Cleanup。

### A* 单线网

多引脚线网先用 MST（Prim）排连接顺序，再在相邻引脚对之间 A*。代价含：基础步长（水平/竖直 1.0，过孔 5.0）、history_cost×history_factor、当前 occupancy 惩罚、以及 net_type==1 时在 M1/M2 的 5.0。启发式用曼哈顿+层差×权重 1.2。路径上的格点记入该线的 occupied_nodes 并更新 occupancy。

### PagedGrid

occupancy、history_cost、is_pin_net_id 和 A* 的 visited 都用 PagedGrid：64×64×层 一块，按需分配，只分配被访问到的块，大规模测例内存从数 GB 降到和实际布线规模相关。块号 `(x>>6, y>>6)`，块内 `(x&63, y&63, layer)`。

### Cleanup

主循环到最大迭代还没消完冲突时，只扫已布线线网的 occupied_nodes 建 cell→nets，找出冲突格和涉及的线网，对这些线网用很大的 history/present（如 50）再 rip 再 route。迭代到无冲突或到 Cleanup 上限。复杂度跟布线长度相关，不做全网格扫描。

### 单线程

作业允许最多 4 线程。这里主循环是单线程：PathFinder 每轮必须按 net_order **顺序** rip+route，后面的线网要看到前面已经更新好的 occupancy；如果多线程并行布不同线网，大家看到的 occupancy 不一致，顺序相当于被打乱，overuse 很难收敛（实测多线程时 overuse 一直抖）。所以没在主循环里按线网分线程；Cleanup、解析、输出也没有再拆多线程。

## 关键实现细节

### 解析

网格：从 grid_info 读 grid_length、max_width、max_height，`width = max_width / grid_length`（整除）。层名按 METAL_LAYER_COUNT 生成 "M1","M2",...。

引脚：bottom_layer 用 c4_name、grid_coord_x/y，除 grid_length 得网格坐标，层 0；top_layer 用 bump_name，层为顶层金属。用 calculatePinMapKey 做键防重（同一格点不同层/类型映射到不同 Key）。

线网：遍历 netlist 的 nets，取 net_name、net_type、pins/bumps，用 pin 名在 bump_info_map 里查坐标和输出层，填 Net 的 bumps、pin_output_layers、pin_original_coords（物理坐标用来写引脚到网格的 L 形段）。net_type 会参与 A* 代价（见下）。

### 坐标与 Key

网格格点 (x, y, layer) 与一维 Key 的换算见上节「网格与 Key」：`Key = (l * grid_height + y) * grid_width + x`，非法层返回 -1。引脚到网格坐标：物理坐标 grid_coord_x/y 除以 grid_length（整除或四舍五入）得到网格 (cx, cy)，层由 C4/Top 固定为 0 或顶层。

### 输出

内部 RouteStep 是起点/终点 Coord（网格坐标+层）。写出时乘 grid_length 变物理坐标，层用 "M1" 等；并在首尾插入引脚物理坐标到网格点的 L 形，保证无锐角。wirelength 只累加同层线段欧氏长度；grid_overuse 是 occupied_nodes 统计后出现次数>1 的格点数；sharp_angles 沿路径三点同层算夹角，本实现一般不产生锐角。格式与 evaluator 期望的 start_grid_coordinate/end_grid_coordinate 一致。

### 布线中的若干设计

- **多引脚线网**：先用 Prim MST 按引脚间曼哈顿距离确定连接顺序，再在相邻点对之间做 A*，以减小总线长。
- **net_type**：`net_type==1` 的线网在 A* 中对 M1/M2（layer 0、1）加额外代价（config 中 NET_TYPE1_LAYER_PENALTY），引导走高层。
- **Cleanup**：只扫已布线线网的 occupied_nodes 建 cell→nets 反向索引；按“涉及冲突格数”对线网排序，优先重布冲突多的线网，再用高 history/present 做 rip+route，迭代至无冲突或达上限。

### 配置（config.hpp）

金属层数、PagedGrid 的 BLOCK_SHIFT（6）、MAX_PATHFINDER_ITERATIONS（50）、MAX_CLEANUP_ITERATIONS（100）、history/present 初值和每轮变化、Cleanup 的 50.0、A* 启发式 1.2、过孔代价 5.0、NET_TYPE1_LAYER_PENALTY 5.0 等都在 config.hpp，改用例时主要改这里。

## 实现脉络（v1～v6）

**v1**：单文件 main.cpp，JSON 解析 + 简单 rip-up & reroute，L 形优先、部分 A*。问题：grid_overuse 很大（十几万），不满足“每格一根线”。

**v2**：扩大 A* 使用范围、多轮迭代。overuse 有所下降但仍很高，且在原始分辨率下跑，耗时长（数小时）。

**v3**：网格粗化（用 grid_length 把物理坐标除成粗网格），在粗网格上做单线程 PathFinder，6 邻域 A*，再加 Cleanup。overuse 能压到 0，但大规模测例内存和速度仍吃紧。

**v4**：A* 的 best_g/parent 复用（AStarCache）、Cleanup 用 cell→nets 反向索引代替全量扫描、MST 多引脚排序等。稳定性和性能进一步改善。

**v5**：参考 [FZU](https://openatom.tech/opendacs2025_competition_4/bade92f99153e0a93f7dda6bfb98e2b5) 的实现，引入 PagedGrid（64×64 块按需分配）替代全平面数组，A* 状态也用 PagedGrid；轻量 Cleanup 只扫已布线单元；参数和工程化整理到 config。大规模用例（如 C4M4）可在合理时间和内存内跑通。

**v6**：尝试在主循环里按线网分多线程并行 rip+route，实测 overuse 不收敛；改为单线程并去掉多线程相关代码，在文档中说明原因（顺序依赖）；并整理提交用文档。

当前代码对应 v5 的架构。

## 总结与参考文献

### 总结

程序完成了作业三点：读入网表并转成布线结构、读入基板 grid 与引脚、在基板上完成布线并输出规定 JSON。实现是 PathFinder 迭代 + A* 单线网 + PagedGrid + 轻量 Cleanup，单线程主循环以保证收敛，无短路断路、无单格多线、无锐角。

### 相对 FZU 参考实现的改进

本实现 (v5) 参考了 [FZU](https://openatom.tech/opendacs2025_competition_4/bade92f99153e0a93f7dda6bfb98e2b5) 的实现（单文件 `router.cpp`，PagedGrid + PathFinder）。在保留其 PagedGrid 与 PathFinder 主循环思路的基础上，做了以下改进：

1. **模块化与配置**：拆分为 main、parser、pathfinder、output、config、types、paged_grid 等模块，便于维护与阅读；JSON 解析使用库（如 nlohmann/json）替代手写解析器。所有可调参数集中在 `config.hpp`（金属层数、迭代上限、Cleanup 因子、A\* 启发式与过孔代价等），换用例时只改配置即可。
2. **多引脚线网**：MST 连接顺序。FZU 按网表中引脚顺序逐段 A\*（bumps[i]→bumps[i+1]）。本实现对多引脚线网先用 Prim 求 MST 确定连接顺序，再在相邻点对之间 A\*，有利于减小总线长。
3. **Cleanup 阶段**：FZU 仅做 PathFinder 主循环，若到最大迭代仍有 overuse 则直接结束。本实现在未收敛时进入 Cleanup：仅扫已布线单元的 occupied_nodes 建 cell→nets 反向索引，对冲突线网施加高 history/present 再拆再布，多数情况下可把剩余冲突消掉，保证 grid_overuse=0。
4. **输出与可观测性**：除写出 output.json 外，控制台输出 wirelength、disconnected_nets、invalid_segments、grid_overuse、sharp_angles，以及总运行时间（秒/分钟/小时），便于与评测脚本对照和做性能分析。
