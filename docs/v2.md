# Router v2 优化记录

本文档记录针对 `grid_overuse_count` 问题所做的优化尝试、结果与经验教训。

---

## 一、初始问题

### 1.1 评测结果（v1 版本）

**C2IO1 测试用例**（700 条线网，网格 4560×3840）：

```json
{
  "summary": {
    "pass": false,
    "disconnected_nets_count": 0,
    "invalid_segments_count": 0,
    "grid_overuse_count": 173745,  // 主要问题
    "sharp_angles_count": 0
  },
  "wirelength": 857942.9819,
  "layers_count": 4
}
```

**核心问题**：`grid_overuse_count: 173745`，违反「每个 grid 走不超过 1 根线」的硬约束，导致 `pass: false`。

### 1.2 v1 实现方式

- **布线策略**：L 形路径优先，短线网（dist ≤ 250）使用 A*，长线网用 L/绕行
- **Rip-up and Reroute**：6 轮迭代，逐条拆、逐条布
- **中间层支持**：4 层（Top, M1, M2, Bottom）
- **问题**：局部优化难以解决全局拥塞

## 二、第一轮优化（局部改进）

### 2.1 优化内容

#### 1. 扩大 A* 使用范围

**改动**：

- **原先**：只有 `dist <= 250` 且 `margin=35` 时使用 A*
- **优化后**：
  - `dist <= 400`：使用 `margin=70`
  - `dist <= 900`：使用 `margin=55`
  - 在**初次布线**就尽量走空位，从源头减少 overuse

**代码位置**：`route_two_same_layer` 函数

```cpp
// 1) 优先 A*：中短距离用 A* 从源头避让
int astar_margin = (dist <= 400) ? 70 : (dist <= 900) ? 55 : 0;
if (astar_margin > 0) {
    std::vector<GridPoint> apath = astar_same_layer(a, b, occupancy, max_x, max_y, astar_margin);
    if (!apath.empty()) return path_to_segments(apath);
}
```

#### 2. Rip-up and Reroute 加强

**改动**：

- **重布顺序**：按每条线网参与的 overuse 格点数**降序排序**，先拆、先重布冲突最多的线网
- **重布轮数**：6 → 12（后调整为 10-14）
- **重布策略**：先试 A*（margin=120），再试 BFS（margin=140），提高在拥挤区域找到无冲突路径的机会

#### 3. 多引脚线网用 MST 顺序连接

**改动**：

- **原先**：多 pin 线网按 pin 顺序链式连接
- **优化后**：按 **MST（Prim 算法）** 顺序连接，缩短总长、减轻拥挤

#### 4. 2 拐点绕行范围扩大

**改动**：

- `try_detour_same_layer` 的 offset 上限：80 → 100

### 2.2 优化结果

**C2IO1 测试结果**：

```json
{
  "summary": {
    "pass": false,
    "disconnected_nets_count": 0,
    "invalid_segments_count": 0,
    "grid_overuse_count": 163572,  // 从 173745 降到 163572
    "sharp_angles_count": 0
  },
  "wirelength": 845139.5554,
  "layers_count": 4
}
```

**效果**：

- `grid_overuse_count` 从 **173745** 降到 **163572**（减少约 6%）
- 但仍**未达到 0**，`pass` 仍为 `false`
- 运行时间：约 2-3 分钟（可接受）

**问题**：

- 改善有限，仍无法彻底消除 overuse
- 局部优化难以解决全局拥塞问题

## 三、第二轮优化（PathFinder 协商拥塞 + 多线程）

### 3.1 优化思路

基于互联网搜索和学术论文，引入 **PathFinder（协商拥塞路由）** 算法：

- **核心思想**：允许暂时 overuse，通过 cost 函数驱赶拥塞
- **Cost 函数**：`cost = 1 + history * usage`
  - `history`：历史拥塞惩罚（每次迭代后更新）
  - `usage`：当前占用次数
- **迭代过程**：
  1. 全拆所有线网
  2. 打乱线网顺序
  3. 用 congestion-aware A* 重新布线（允许暂时 overuse）
  4. 更新 history cost（overuse 的格点 history++）
  5. 重复直到 overuse == 0 或达到迭代上限

### 3.2 实现内容

#### 1. 协商拥塞 A* 算法

**新增函数**：`astar_congestion`

```cpp
std::vector<GridPoint> astar_congestion(
    const GridPoint& a, const GridPoint& b,
    const std::unordered_map<std::string, int>& usage,
    const std::unordered_map<std::string, int>& history,
    int max_x, int max_y, int margin
) {
    // Cost 函数：1 + history * usage
    auto cell_cost = [&usage, &history](const std::string& k) -> long long {
        int u = 0, h = 0;
        auto iu = usage.find(k); if (iu != usage.end()) u = iu->second;
        auto ih = history.find(k); if (ih != history.end()) h = ih->second;
        return 1 + (long long)h * (long long)u;
    };
    // A* 搜索，允许走已占用格点但惩罚
    // ...
}
```

**特点**：

- 允许暂时 overuse，但通过 cost 函数惩罚
- 在原始分辨率（4560×3840）上运行，状态空间巨大

#### 2. PathFinder 主循环

**新增函数**：`run_pathfinder`

```cpp
std::pair<json, int> run_pathfinder(
    const std::vector<std::pair<std::string, std::vector<GridPoint>>>& nets_with_pins,
    int max_x, int max_y,
    // ...
) {
    std::unordered_map<std::string, int> usage;  // 当前占用
    std::unordered_map<std::string, int> history; // 历史拥塞

    for (int iter = 0; iter < MAX_ITER; ++iter) {
        // 1. 全拆
        usage.clear();

        // 2. 打乱顺序
        std::shuffle(net_order.begin(), net_order.end(), rng);

        // 3. 重新布线（允许暂时 overuse）
        for (int net_id : net_order) {
            route_two_pins_congestion(..., usage, history, ...);
        }

        // 4. 计算 overuse
        int overuse = count_overuse(usage);

        // 5. 更新 history（overuse 的格点 history++）
        for (auto& [cell, u] : usage) {
            if (u > 1) history[cell]++;
        }

        if (overuse == 0) break;
    }

    return {output_json, overuse};
}
```

**参数调整**：

- `max_pf_iter`：从 22 降到 14（减少迭代次数以控制时间）
- `margin`：从 `min(250, 80 + dist/10)` 降到 `min(150, 50 + dist/15)`（缩小搜索范围）

#### 3. 多线程并行尝试不同顺序

**实现**：4 线程并行运行 PathFinder，使用不同的线网顺序：

- 顺序 1：按 span 升序（短线网优先）
- 顺序 2：按 span 降序（长线网优先）
- 顺序 3：按中心 x 坐标
- 顺序 4：按中心 y 坐标

取 overuse 最小的结果。

**代码**：

```cpp
std::vector<std::pair<json, int>> results(4);
std::vector<std::thread> threads;
threads.emplace_back([&]() { results[0] = run_pathfinder(nets_with_pins, max_x, max_y, order_span_asc); });
threads.emplace_back([&]() { results[1] = run_pathfinder(nets_with_pins, max_x, max_y, order_span_desc); });
threads.emplace_back([&]() { results[2] = run_pathfinder(nets_with_pins, max_x, max_y, order_cx); });
threads.emplace_back([&]() { results[3] = run_pathfinder(nets_with_pins, max_x, max_y, order_cy); });
for (auto& t : threads) t.join();

int best_idx = 0;
for (int i = 1; i < 4; ++i) {
    if (results[i].second < results[best_idx].second) best_idx = i;
}
```

**问题**：

- 4 线程各跑 14 轮 PathFinder，总计算量巨大
- 在原始分辨率（4560×3840）上运行，A* 搜索空间极大
- 单次 A* 可能需要探索数百万个节点

#### 4. 层数调整

**改动**：

- 从 4 层（Top, M1, M2, Bottom）改为 **2 层**（Top, Bottom）
- **原因**：尝试减少状态空间，但效果不明显

### 3.3 优化结果

**C2IO1 测试结果**（PathFinder + 4线程）：

```json
{
  "summary": {
    "pass": false,
    "disconnected_nets_count": 0,
    "invalid_segments_count": 0,
    "grid_overuse_count": 134425,  // 从 163572 降到 134425
    "sharp_angles_count": 0
  },
  "wirelength": 678048.31,
  "layers": ["bottom", "top"],
  "layers_count": 2
}
```

**效果**：

- `grid_overuse_count` 从 **163572** 降到 **134425**（减少约 18%）
- PathFinder 算法有效，能逐步减少 overuse
- **运行时间超过 4 小时**，无法在合理时间内完成
- 仍未达到 `grid_overuse_count: 0`

**问题分析**：

1. **状态空间过大**：在原始分辨率（4560×3840）上运行 PathFinder，A* 需要探索的节点数巨大
2. **多线程放大计算量**：4 线程各跑 14 轮，总计算量是单线程的 4 倍
3. **收敛速度慢**：虽然 PathFinder 能减少 overuse，但在如此大的状态空间下收敛极慢

## 四、优化总结

### 4.1 优化效果对比

| 版本                               | grid_overuse_count | layers_count | 运行时间          | 主要方法                 |
| ---------------------------------- | ------------------ | ------------ | ----------------- | ------------------------ |
| **v1 初始**                  | 173745             | 4            | ~3 分钟           | L 形 + 简单 RRR          |
| **第一轮优化**               | 163572             | 4            | ~3 分钟           | 扩大 A* + 改进 RRR + MST |
| **第二轮优化（PathFinder）** | 134425             | 2            | **>4 小时** | PathFinder + 4线程       |

### 4.2 关键优化点

1. **扩大 A* 使用范围**：从源头避让，减少初始冲突
2. **按冲突数排序重布**：优先处理冲突最多的线网
3. **MST 顺序连接**：多引脚线网更优连接顺序
4. **PathFinder 协商拥塞**：允许暂时 overuse，通过 cost 函数迭代驱赶
5. **多线程并行尝试**：4 种线网顺序并行，取最优结果

### 4.3 经验教训

1. **PathFinder 有效但需要优化**：

   - PathFinder 算法本身能有效减少 overuse
   - 但在原始分辨率上运行，状态空间过大，导致运行时间过长
2. **多线程的权衡**：

   - 多线程能尝试不同顺序，找到更好的解
   - 但会放大计算量，在状态空间大的情况下反而拖慢整体速度
3. **状态空间是关键**：

   - 在原始分辨率（4560×3840）上运行，A* 搜索空间巨大
   - 单次 A* 可能需要探索数百万个节点
   - **需要缩小状态空间**才能提高效率
4. **层数的影响**：

   - 从 4 层降到 2 层，状态空间从 `4560×3840×4` 降到 `4560×3840×2`
   - 但改善有限，因为网格分辨率本身太大

### 4.4 核心问题

**根本原因**：在**原始分辨率**（4560×3840）上运行 PathFinder，状态空间过大：

- 网格节点数：`4560 × 3840 × 2 = 35,020,800` 个节点
- A* 搜索空间：每次搜索可能需要探索数百万个节点
- 4 线程 × 14 轮：总计算量巨大

**解决方案**：需要**缩小状态空间**，例如：

- **网格粗化（Grid Coarsening）**：将坐标除以 `grid_length`（如 5），在粗网格上布线
- 粗网格节点数：`912 × 768 × 2 = 1,400,832` 个节点（约为原来的 1/25）
- 状态空间大幅缩小，A* 搜索更快，PathFinder 迭代更快

## 六、后续方向

虽然 PathFinder 解决了算法问题，但运行时间仍然过长。后续引入：

1. **网格粗化（Grid Coarsening）**：将搜索空间从原始分辨率降到粗网格，大幅减少状态空间
2. **PagedGrid 稀疏网格**：大网格下按需分配内存
3. **更高效的 PathFinder 实现**：优化数据结构、减少重复计算
4. **单线程 PathFinder**：去掉多线程，专注于单线程优化
